起因

家里的WiFi总是过一点时间就会很卡，让人不爽！

网络布局

家里的网络布局和绝大多素普通家庭类似， 一个200M的电信宽带接上一个路由器

排障流程

从上文的拓扑图看，设备接入互联网慢，可能是两个组件有问题：电信光猫或者是我们的路由器，当然也有可能两者同时故障。

  1. 路由器问题？

通过对比台式电脑和移动设备之间的网速，发现其并不存在明显差距——都是龟速，而路由器是新买不久的支持5G频率的千兆路由器，先大胆排除是路由器故障。

  1. 电信光猫

在台式电脑上（这点很重要，保证了整个物理链路的物理带宽为千兆) 使用上海电信官网的测试界面进行测速结果只有1MB左右(具体数值忘了截图了)，距离200Mb/8≈25MB 还有很大差距。

现在初步认定是电信光猫的问题，那么如何解决这个问题呢？

方案1： 电信客服投诉，保修？

根据平时经验，网速有时能恢复，工人上门维修的时候不一定能复现。另外通过微信公众号的自检，电信认定为设备正常。 工人最多也就过来重启下，然后测试正常就算完了。 这显然不能解决问题

方案2： 其它？

比如换移动、联通？嗯，这是在搞笑。

在电信的测速界面看到一个非常有趣的提醒，大意是说“现在测速结果偏慢，但是要在重启网关之后才能保证测速准确哦”

于是萧先生抱着试一试态度，重启了路由器。您别说，还真有效！

于是，现在问题转变成，如何自动重启光猫

通过路由器的剩余计算能力闲时自动重启光猫

如何自动重启光猫

家里的光猫可以通过web界面重启， 从这点出发，我们只需要抓包破解重启通信协议即可达成目标。

家里的光猫管理界面如下图所示：

通过Chrome自带的HTTP抓包结果如下

登录请求

HTTP GET http://192.168.1.1/login.cgi?username=useradmin&psd=<登录密码>

登录后，你会发现web界面不断地发送post请求和光猫内的服务端进行数据同步

HTTP POST http://192.168.1.1/gatewayManage.cmd?timeStamp=1577285343462

其中POST的body以表单形式发送，如：jsonCfg=%7B%22RPCMethod%22%3A%22Post1%22%2C+'ID'%3A'123'%2C+%22Parameter%22%3A'eyJDbWRUeXBlIjoiSEdfQ09NTUFORF9SRUJPT1QifQ%3D%3D'%7D

URL decoded 之后为：jsonCfg={"RPCMethod":"Post1", 'ID':'123', "Parameter":'eyJDbWRUeXBlIjoiSEdfQ09NTUFORF9SRUJPT1QifQ=='}

通过观察多个请求，可以发现请求jsonCfg中的字段 RPCMethod和ID都是固定不变的， 而Parameter是变化的。通过观察多个Parameter实例，可以注意到其是Base64编码的（因为有以=结尾的，=是base64编码的padding）。

eyJDbWRUeXBlIjoiSEdfQ09NTUFORF9SRUJPT1QifQ== base64 decoded 之后为 {"CmdType":"HG_COMMAND_REBOOT"}

这正是我们想要的请求

URL中还有一个query参数为timeStamp, 这个很直观的知道是时间戳，通过实验可知这个参数用来拒绝过期的请求（多长时间被视作过期没有测试）

按照常识，HTTP header中应该有登录之后的session id，然而并木有！！！

按照这个尿性，那header中的 Referer 肯定也不会验证了。于是萧先生大胆猜测一个重启指令只需要发送如下简单的post请求

1
2

# 注意更新timeStamp
curl 'http://192.168.1.1/gatewayManage.cmd?timeStamp=1577286579429'  --data $'jsonCfg=%7B%22RPCMethod%22%3A%22Post1%22%2C+\'ID\'%3A\'123\'%2C+%22Parameter%22%3A\'eyJDbWRUeXBlIjoiSEdfQ09NTUFORF9SRUJPT1QifQ%3D%3D\'%7D'

执行后打印的http body如下，其中 return_Parameter base64解码后为{ "CmdType": "HG_COMMAND_REBOOT", "SequenceId": null, "Status": "0" }。 0 也就是 OK 咯？

1

{ "ID": 123, "_Source": 0, "Result": 0, "return_Parameter": "eyAiQ21kVHlwZSI6ICJIR19DT01NQU5EX1JFQk9PVCIsICJTZXF1ZW5jZUlkIjogbnVsbCwgIlN0YXR1cyI6ICIwIiB9" }

执行后，发现光猫的管理界面打不开了（可能是在重启），过一会等等再次登录之后，通过查看光猫的运行状态（运行时间），证实了刚才的重启是有效的！
