goroutine 是如何调度的？为什么可以在单核上创建百万级goroutine

GMP 调度模型

采用GMP模型来调度，即将G个goroutine，分为P个processor 分组，调度到底层的M个OS thread之上。P为逻辑概念，用户可配，默认为CPU核数。

  • G 每个goroutine维护了运行状态及栈信息
  • P 每个P都有一个local run queue。此外系统也有全局的run queue。 任何一个可运行的go routine必须保存到P的局部队列或者全局队列中
  • M 必须和一个P绑定才可以运行goroutine。当M在运行goroutine即将进入syscall之前，会将P传递给另外一个M。

参考资料

  • Illustrated Tales of Go Runtime Scheduler. 介绍go为什么要采用三层调度。

首先，为什么不是一层调度（即只用os thread)? 因为context 需要保存的信息多，并且context switch开销也大

其次，为什么不是两层调度？ 两层调度M:N模型，M个OS thread负责调度N个go routine。

  1. 如果所有的OS thread都从一个全局的队列中获取并执行go routine，那么这个共享区域需要一个锁保护，不可横向扩展。并且，如果一个goroutine不断地在不同的OS thread上执行，那么这个go routine很可能在不同的CPU之间来回偏移，不能利用CPU缓存（L1~L2 cache)带来的性能提升
  2. 如果每个OS thread都有一个局部的队列，也同样有问题。OS thread在进行syscall的时候会被block，此时局部队列上的其它goroutine不能被执行，为了保证CPU被充分利用，其它局部队列为空的空闲的OS thread需要从别的OS thread之上“偷”goroutine来执行，遍历每个OS thread的局部队列是一个O(M)的过程，而M通常比希望的实际运行并行数量要大。比如一个4核CPU，我们可能有100个OS thread，但是实际并行（注意不是并发）的goroutine数量最多只有4个。

  • Gao Chao的《go runtime scheduler》slides

    包含了Local Run Queue， global run queue的图示，以及work sharing 以及 work stealing的算法。

  • YouTube《Go scheduler: Implementing language with light weight concurrentcy》by Dmitry VyuKov

    详细地解释go scheduler为什么要这样设计，比前面两个资料有更多的细节。

    • 比如在P的FIFO 本地队列之前，还有一个大小唯一的LIFO队列，这样能做到更好的locality
    • 怎么保证global run queue中的不会被饿死？每61次调度循环之后，一定要查询一个global run queue
    • goroutine的stack是怎么组织的？为什么goroutine的开销这么低。growable stack一开始很小，按需增大。在GC的时候如果stack有很多闲置空间再回收

channel 与select是怎么工作的

Chanel 比较直观：

  1. 有一个锁保护内部结构，发送/接收之前需要获取这个锁
  2. 有一个环状的buffer（可能大小为0）
  3. 一个recvq保存阻塞在接收的goroutines
  4. 一个sendq保存阻塞在发送的goroutines
  5. 在发送数据时，如果此时recvq有阻塞的goroutine，那么数据直接从发送goroutine “复制”到recvq队列中的第一个goroutine中，并将此goroutine变成runnable供调度。反之亦然

select:

  1. 将涉及到的channel按照内存地址排序，获取所有channel的锁。排序获取是为了防止死锁
  2. 随机地确定一个轮询顺序，依次判断每个case，有满足的则中断，不再轮询后续case
  3. 如果没有任何一个case满足，那么将当前goroutine添加到对应channel对应的recvq/sendq中
